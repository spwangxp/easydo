# EasyDo v0 功能文档

本文档描述 EasyDo 项目的已实现功能，包括前端页面、后端 API 和执行器 Agent 的功能作用、条件与约束。

---

## 一、项目架构概览

### 1.1 技术栈

| 模块 | 技术栈 |
|------|--------|
| 前端 | Vue 3 + Composition API + Element Plus + Vite + Pinia |
| 后端 | Go 1.21 + Gin + GORM + MySQL + Redis |
| 执行器 | Go + 心跳机制 + 任务调度 |

### 1.2 项目结构

```
easydo3/
├── easydo-frontend/     # 前端项目
├── easydo-server/       # 后端服务
├── easydo-agent/        # 执行器 Agent
└── doc/                 # 文档目录
```

## 二、前端功能模块

### 2.1 认证模块

#### 登录页面 (`/login`)

**功能作用**：提供用户身份认证入口，支持多种登录方式。

**条件**：
- 用户已注册账号
- 具备有效的用户名/密码
- 第三方登录需完成对应的应用配置

**约束**：
- 登录成功后生成 JWT Token
- Token 有有效期限制
- 未登录用户无法访问需要认证的页面

## 三、后端 API 接口

### 3.1 认证模块 (Auth)

| 接口路径 | 方法 | 功能 | 请求参数 | 响应数据 |
|---------|------|------|---------|---------|
| `/api/auth/login` | POST | 用户登录 | `username`, `password` | token, userInfo |
| `/api/auth/register` | POST | 用户注册 | `username`, `password`, `email` | userInfo |
| `/api/auth/logout` | POST | 用户登出 | 无 | success |
| `/api/auth/profile` | GET | 获取用户信息 | 无 | userInfo |
| `/api/auth/refresh` | POST | 刷新Token | `refresh_token` | newToken |

**条件与约束**：
- 登录/注册需要提供有效的用户名和密码
- 用户名唯一，长度限制
- Token 有效期限制
- 刷新Token需要有效的refresh_token

## 四、执行器 Agent

### 4.1 核心功能模块

#### Agent 核心模块 (core/)

| 模块 | 文件 | 功能 |
|------|------|------|
| Agent管理 | `agent.go` | Agent生命周期管理、心跳检测、状态监控 |
| 任务处理 | `task_processor.go` | 任务处理引擎，支持任务队列、优先级调度 |
| 任务管理 | `task_manager.go` | 任务创建、执行、监控、完成管理 |

| 处理器 | 文件 | 功能 |
|--------|------|------|
| 执行处理器 | `execute_handler.go` | 命令执行、进程管理 |
| 脚本处理器 | `script_handler.go` | 脚本解释器支持（Shell、Python、Bash等） |
| API处理器 | `api_handler.go` | HTTP API调用、接口测试 |

#### 数据模型 (models/)

| 模型 | 文件 | 字段 |
|------|------|------|
| 任务 | `task.go` | ID、标题、描述、状态、优先级、执行参数等 |

### 4.3 任务处理流程

#### 4.3.1 任务获取流程

```
1. Agent启动 → 连接Server注册
2. 定期心跳 → 向Server报告状态
3. 拉取任务 → 从Server获取待执行任务
4. 本地队列 → 任务加入本地队列
5. 任务调度 → 按优先级分配执行
```

#### 4.3.2 任务执行流程

```
1. 任务预处理 → 参数解析和验证
2. 资源检查 → 检查执行环境和资源
3. 执行引擎 → 调用对应的处理器执行
4. 实时监控 → 监控执行状态和进度
5. 结果收集 → 收集执行结果和日志
6. 完成上报 → 向Server上报执行结果
```

#### 4.3.3 异常处理机制

| 异常类型 | 处理方式 |
|---------|---------|
| 执行超时 | 强制终止任务，标记超时 |
| 执行失败 | 根据配置决定是否重试 |
| 资源不足 | 等待资源释放后重试 |
| 通信中断 | 尝试重连，任务状态同步 |

---

### 4.4 配置项说明

| 配置项 | 描述 | 默认值 | 约束 |
|--------|------|--------|------|
| `server_address` | Server地址和端口 | 必填 | 格式：host:port |
| `agent_id` | Agent唯一标识 | 必填 | 与Server注册一致 |
| `heartbeat_interval` | 心跳间隔 | 30s | 最小10s |
| `task_poll_interval` | 任务拉取间隔 | 10s | 最小5s |
| `max_concurrent_tasks` | 最大并发任务数 | 4 | 最小1 |
| `exec_timeout` | 默认执行超时时间 | 600s | 最小60s |
| `retry_times` | 失败重试次数 | 3 | 最小0 |

---

### 4.5 与 Server 交互

#### 4.5.1 注册与认证

| 阶段 | 描述 | 约束 |
|------|------|------|
| 启动注册 | Agent启动时向Server注册 | 需要有效的Agent ID |
| 身份验证 | 提供认证信息 | 认证失败拒绝连接 |
| 权限获取 | 获取任务队列访问权限 | 审批通过后才可获取任务 |

#### 4.5.2 任务流转

```
Server → 任务分配 → Agent拉取 → 本地调度 → 执行
                                              ↓
Server ← 结果上报 ← 执行完成 ← 状态同步 ← ← ← ←
```

**交互约束**：
- 所有通信需要认证
- 任务拉取受频率限制
- 结果上报必须在任务完成后及时执行
- 心跳超时Server会标记Agent离线

## 六、通用约束与规范

### 6.1 认证与权限

| 规则 | 描述 |
|------|------|
| JWT认证 | 所有API请求需要有效的JWT Token（除登录/注册） |
| 角色权限 | 分为admin和user角色，管理员拥有所有权限 |
| 资源所属 | 用户只能操作自己创建的资源（除管理员） |
| 敏感操作 | 删除操作需要二次确认 |

### 6.2 数据验证

| 规则 | 描述 |
|------|------|
| 用户名格式 | 3-50个字符，仅字母数字下划线 |
| 邮箱格式 | 有效的邮箱地址格式 |
| 密码强度 | 至少8位，包含字母和数字 |
| 文件大小 | 单文件最大100MB |
| 请求频率 | 有限流保护 |

### 6.3 响应格式

所有API返回统一JSON格式：

```json
{
    "code": 200,           // 状态码：200成功，400参数错误，401未认证，403无权限，500服务器错误
    "message": "success",  // 状态消息
    "data": {              // 业务数据
        // ...
    }
}
```

### 6.4 中间件功能

| 中间件 | 功能 |
|--------|------|
| JWT Middleware | 请求认证和Token验证 |
| Logger Middleware | 请求日志记录 |
| CORS Middleware | 跨域资源共享 |
| Rate Limit Middleware | 请求频率限制 |
